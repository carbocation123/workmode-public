# 科研协作与文件系统纪律

## 零、教程优先级

- 本项目的工作成果是教学道具；工作不是目的，演示才是目的。
- AI 必须遵守 `TUTORIAL_AI_GUIDE.md` 的主持协议和 `notes/demo-state.md` 的单元顺序。
- 每次用户回复最多完成一个演示单元；“开始、可以、继续、按建议执行”不得被扩大解释为后续章节授权。
- 每个单元先说明演示目标、已看到的文件、准备动作、观察重点和本步边界；完成后复盘并停下。

## 一、基本原则

- 先读取项目根 README 和距离目标最近的目录 README，再开始操作。
- 没有实际读取的文件、数据或论文原文，不得当作事实引用。
- 明确区分「来源事实」「作者观点」「数据观察」和「AI 推断」。
- 关键歧义会改变科研结论、文件结构或处理方法时，先询问用户。
- 用户负责最终科学判断；AI 负责检索、整理、分析、起草和一致性维护。

## 二、文件系统纪律

- 项目根 README 必须与硬盘真实结构一致。
- 主干目录必须有 README，说明用途、内容、输入、输出、权威来源和关联位置。
- 修改前搜索目标文件、路径、编号和结论被哪些说明、索引、日志、脚本和报告引用。
- 新增、移动、重命名或删除主干内容时，同步更新父级说明、局部说明、索引、链接和脚本路径。
- 关键关系必须能够双向追溯；不能只更新一端。
- 如果无法完成全部联动，不留下半更新状态，应停止并说明阻塞。

## 三、数据与文献纪律

- `data/raw/` 是不可修改的原始数据区：不覆盖、不删除、不重命名。
- 派生数据写入 `data/processed/`，并记录原始文件、脚本、参数、异常处理和生成时间。
- 搜索结果和网页摘要只负责发现文献，不能代替 PDF 原文。
- 文献事实应注明页码、章节、Figure 或 Table；找不到时标记「待核实」。
- 缺失值、重复记录和异常值必须报告，不得静默修补。
- 读取本地文件不等于允许外传。工具会向 MinerU、模型 API 或其它第三方发送 PDF、正文、数据时，必须先说明服务、上传对象和风险，并为每类传输取得用户明确同意。
- API key 只放本机配置或环境变量，不写入项目说明、日志、报告、示例配置或聊天正文。

## 四、修改与验证

- 修改已有文件前先读取当前内容；局部修改优先精确编辑。
- 写入前简要说明准备修改什么以及会联动哪些位置。
- 删除文件、覆盖原始数据或进行破坏性操作必须获得用户明确许可。
- 完成后复读关键文件，并检查旧路径、失效链接、索引和来源链。
- 没有工具结果或验证证据，不得声称「已写入」「已分析完」或「已修复」。

## 五、结果汇报

任务完成后简要说明：

1. 修改了什么；
2. 使用了哪些来源；
3. 做了什么验证；
4. 还存在哪些不确定项。
