# 特化 Workmode 模块开发规范

## 定位

特化模块不是第二个聊天软件。它由同一套 Workmode 交互内核、一个固定结构项目、一组最小领域工具和一个领域前端投影组成：

```text
Workmode session / JSONL / context / compaction / stop
                         │
                  project manifest
                         │
        fixed files ─ domain tools ─ frontend projection
```

文献库是第一套落地实现。后续数据分析、实验规划或写作模块应复用相同分层，不再复制 session、模型循环、工具事件、压缩器、设置或项目记忆。

当前文献参考实现的领域后端位于 `backend/app/literature_*.py`，活动前端位于 `frontend/src/literature/`。应用启动先进入功能大厅；大厅不再渲染单独的解释性介绍面板，两张入口卡跨越中间内容区，但使用受控高度并在顶栏与底栏之间垂直居中；底栏继续固定在第四行。因此删减文案不会纵向拉长入口卡，也不会让已导入的签名皮肤把底栏排进主内容区。窄屏下取消固定高度，入口卡恢复自然纵向排列。科研工作台说明为“定制你自己的AI工作流”，文献智库标记为“文献特化模块”。大厅通过 `/api/work/literature-projects` 恢复或初始化固定结构项目，并把同源 WebView 导航到正式 Vite 多页面入口 `/literature/index.html`。完整科研工作台是大厅的另一个同级入口，使用 `?surface=workbench`；Tauri 仍只打包一份 `frontend/dist`，不启动第二个前端服务。

跨模块导航只属于功能大厅，不在完整工作台活动栏重复堆叠。特化页面需要提供全局设置入口时，应导航到共享设置页并携带返回来源，不能复制模型、外观、更新或认证配置表单。特化页面的中性控件、输入、浮层和文档表面必须先消费 `themeContract.css` 的共享 `--ui-*` 角色；领域页面可以再用自己的变量前缀建立语义别名，但不能重新定义一套明暗默认值，也不能在组件规则中写死仅适合暗色或亮色的颜色。这样 foundation 与签名皮肤无需逐控件补丁即可切换。

特化页面的固定标题栏只保留定位信息和真实可操作控件，不增加不承载状态或功能的拟人化口号。文献智库对话顶栏只显示 `RESEARCH CONVERSATION` 模块标识，以及 session、上下文、记忆和笔记控件。

## Manifest

每个特化项目在根目录放一个稳定 manifest。文献项目当前为：

```json
{
  "project_type": "literature-library",
  "schema_version": 1,
  "tool_profile": "literature",
  "frontend_projection": "literature-library"
}
```

- `project_type`：决定项目结构策略和领域服务。
- `schema_version`：决定校验与未来迁移入口。
- `tool_profile`：决定实际发送给模型的工具白名单。
- `frontend_projection`：决定项目文件如何显示，不改变事实来源。

未知类型或损坏 manifest 不能静默回退成另一种特化模式。

## 五层边界

1. **通用内核**：项目注册、session meta、JSONL、流式模型、工具事件、token 窗口、压缩、停止、模型设置和项目记忆。
2. **项目结构**：固定目录、机器索引、允许的文件类型、权威/派生文件和版本号。
3. **领域工具**：仅暴露用户任务所需的高层操作；工具名、schema、executor 和返回结果必须成套。
4. **项目协议**：根目录 `WORKMODE.md` 用 `@` 引入领域说明，解释结构、证据边界和工具语义。不可破坏的规则必须同时由代码校验。
5. **前端投影**：把权威文件显示成领域熟悉的卡片、筛选器、详情页或图表；不能另存一套长期业务状态。

## 标准生产流水线

新增模式必须按“真实任务 → 固定项目 → 领域工具 → 正式内核 → 前端投影 → 真实验收”的顺序推进。前端页面不是模式本体；即使前端暂时不可用，固定项目中的权威文件也必须保持完整、可读和可迁移。

### 阶段 0：定义最小业务闭环

先用真实用户任务编写 `MODE_SPEC.md`，不要从静态页面或功能清单开始。最少写清：

- 用户要完成的具体任务、输入与最终产物；
- 一次最小闭环的自然语言步骤；
- 原始事实、机器权威数据、用户判断和 AI 推理各自的边界；
- 明确不做的事情；
- 一个可以用于最终验收的真实样例；
- 成功、失败、中止和恢复的判定标准。

只有闭环可以用真实样例完整走通，才进入项目结构设计。

### 阶段 1：设计固定项目结构

先确定文件系统，再写工具和界面。此阶段必须交付：

- 根 manifest 及稳定的 `project_type`、`schema_version`、`tool_profile`、`frontend_projection`；
- 固定目录、允许的文件类型、权威文件、用户可编辑文件和派生产物清单；
- 文件重复、命名碰撞、损坏数据和版本迁移策略；
- `WORKMODE.md` 及其通过 `@` 引入的领域协议；
- 目录、schema、引用关系、相对路径和原子写入的代码约束。

Windows 上的项目归属校验必须先把根目录和目标路径规范化为真实路径，再计算相对路径；同一目录可能同时以用户名长路径和 8.3 短路径出现，不能直接用未经规范化的字符串判定越界。

提示词负责解释协作方式，代码负责保证结构不可被破坏。不能把“不要修改某目录”只写在 Prompt 中。

推荐的抽象结构是：

```text
项目根目录/
├─ <mode>-project.json       模式身份与 schema 版本
├─ WORKMODE.md               固定 @注入入口
├─ <MODE>_PROJECT.md         领域结构、证据和工具纪律
├─ 原始文件/
├─ 权威机器数据
├─ 用户可编辑内容/
├─ 派生产物/
└─ exports/
```

### 阶段 2：定义最小领域工具画像

每个模式默认只加载完成闭环必需的领域工具，不自动继承工作台、Web、Shell、Python、通用文件、memory 或 plan 工具。每个工具必须同时具备并保持一致：

- 工具名和用途边界；
- JSON schema；
- executor；
- 面向模型的描述与 Prompt 调用说明；
- 成功、失败和取消结果契约；
- 路径、项目身份、字段、冲突与原子写入测试。

模型调用写工具就代表请求执行该领域操作，不再叠加关键词确认、隐藏 proposal 或“必须先选中对象”等状态机。成功结果至少包含 `operation`、目标 ID、`changed_files`/`changed_paths` 和 `updated_fields`；失败结果包含稳定 `error_code`、可读消息和空的改动列表。批量领域工具应提供单项结果数组并限制并发度，使一个对象失败不会取消整个批次；文献参考实现的 `literature_process(paper_ids)` 上限为 3 路并发。

### 阶段 3：接入正式 Workmode 内核

新模式只能扩展项目识别、工具画像、领域 Prompt、项目事件解释和 `active_context`。以下能力一律复用正式内核，不允许复制：

- session meta 与 JSONL；
- 模型循环与 SSE；
- 工具开始、结果、失败和取消事件；
- token 窗口、上下文选择与压缩；
- 停止生成；
- 项目记忆、模型设置、认证和外观状态。

用户当前勾选的文件、论文、样品或笔记只作为当轮 `active_context` 提高相关性，不能被升级成工具权限条件。

### 阶段 4：实现领域服务与投影 API

领域服务负责读写固定结构，投影 API 只把权威文件转换为前端需要的卡片、筛选项、详情或图表：

```text
权威文件 → 领域服务 → 投影 API → 前端
```

复杂流水线可以由领域服务管理，但不能为它另建聊天 session、上下文、压缩器或长期任务真相源。工具执行后，前端应根据 `changed_paths` 合并刷新受影响的权威投影，并在本轮终止时做一次全量校准；工具返回对象只用于即时反馈，不是持久状态。并发写入同一个机器索引时必须在领域服务中加锁并原子替换，不能依赖 Prompt 或前端串行化。

特化模块的默认用户路径应先对齐熟悉的轻量 AI 交互，再把领域流水线作为按需增强。对象导入、选中或进入当前上下文只产生可追溯的 session system 事件，不应自动解释成“开始处理”。文献参考实现的默认闭环是“确认 PDF 入库 → 用户下一次提问 → `literature_read` 按需读取 → 简洁回答”；MinerU、元数据标准化、客观事实报告和归档只在用户明确要求或轻量读取不能满足证据需求时进入。新模式不能仅因为后端已经具备完整流水线，就强迫普通用户先走完整流水线。

### 阶段 5：接入领域前端

领域前端应采用目标用户熟悉的交互模型，同时直接消费正式 Workmode SSE：

- 用户消息先立即显示；
- 正文增量到达即显示；
- `tool_call_start` 立即建立运行中卡片；
- `tool_result` 按 `tool_call_id` 更新同一卡片；
- 保持“文字 → 工具 → 文字”的真实顺序；
- 用户接近底部时跟随流式输出，主动向上滚动后暂停自动跟随并提供返回最新入口；
- 成功工具结果按 `changed_paths` 即时刷新受影响投影，终端事件再全量校准；
- 完成后从 JSONL 回读校准；
- 多 session、停止、压缩和设置复用共享能力；
- 模式内可创建多个项目时，新建、切换、重命名和安全移除必须在领域前端可达；托管路径由后端分配，旧外部项目不得被强制搬迁；
- 新 session 可以由模式初始化器写入一条正式 assistant 开场白；导入与选择等模式事件作为 JSONL system 消息永久保留，并在实时 SSE 中与工具事件同级可见；
- 不保存第二份 catalog、标签、任务或业务状态。
- 对每一种可创建的持久对象同时提供修改与可恢复删除；领域回收站必须处理权威索引、派生产物、引用和活动 UI 状态，不得靠前端隐藏代替删除。

静态 Demo 可以用于讨论布局，但必须与正式入口隔离，不能保留一套会被误启动的独立后端、模拟消息或永久假数据。

### 阶段 6：同步导航、语义槽位与皮肤

皮肤适配是模式交付的一部分，不是发布后的补丁：

- 在应用级功能大厅注册同级入口；
- 返回大厅与全局设置复用共享导航；
- 中性表面消费 `themeContract.css` 的 `--ui-*`；
- 为页面外壳、领域区域、共享聊天和关键操作暴露稳定 `data-skin-slot` / `data-skin-icon`；
- 不复制主题选择、签名皮肤加载器或恢复状态；
- 至少验收默认暗色、基础亮色、高对比、Neon 结构主题和一套官方签名皮肤；
- 验收窄窗口以及 Windows 125% / 150% 缩放。

### 阶段 7：使用真实项目验收

不能用离线样稿或静态消息代替端到端结果。至少验证：

- 初始化、关闭后重开和 schema 识别；
- 真实文件导入与完整业务闭环；
- 工具成功、失败、取消和超时；
- session 新建、切换、重命名和软删除；
- 特化前端的重命名入口必须调用共享 session update API，并在本地投影中即时更新；不能在领域目录、catalog 或独立 store 里再保存一份标题；
- 长上下文压缩及压缩后的续接；
- 重复内容、目标冲突、半写入防护和损坏数据恢复；
- 创建对象的修改、回收、冲突恢复与不可变历史保留；
- 前端刷新后与权威文件一致；
- 可自动规范化的展示格式不应交给模型反复试错；写工具必须先完成规范化与校验，再原子提交领域文件；
- 源码环境与正式桌面安装包环境。

只有真实样例通过后，README 才能把该能力描述为已交付。

### 阶段 8：发布与持续维护

同步 README、总架构、开发说明、模式专属文档、测试基线、schema 版本和迁移策略。后续修改必须同时检查“固定文件 → 机器索引 → 领域服务 → 前端投影 → 项目协议”是否双向一致，避免只修一个页面或一个 JSON 字段。

## 开发流程

1. 从真实用户任务定义最小闭环和明确的非目标。
2. 先设计项目目录、机器 schema、权威来源、派生文件与迁移版本。
3. 列出完成闭环所需的最小工具集；默认不继承其他模式工具。
4. 为项目识别、工具白名单、路径策略、原子写入和跨层映射先写回归测试。
5. 接入正式 Workmode 工具循环；禁止复制 session、上下文、压缩和停止逻辑。
6. 编写领域 Prompt，并用项目 `WORKMODE.md` 固定注入；核对工具描述与实际 executor。
7. 实现前端投影 API，让前端每次从权威文件重读，而不是把工具返回对象当作永久状态。
8. 接入领域前端；当前选择项作为 `active_context` 写入用户消息元数据，不自动升级成权限门槛。
9. 在应用级功能大厅注册同级入口，并为页面外壳、领域区域和共享聊天组件提供稳定 `data-skin-slot`；不要把特化入口塞进另一个特化模块内部。
10. 直接消费正式 Workmode SSE，使用户消息、正文增量、工具开始/结果和停止状态即时可见；完成后从 JSONL 回读校准，不另建前端持久状态源。
11. 使用真实样例完成端到端验收：新建、重启、工具失败、停止、压缩、session 切换、文件冲突和恢复。
12. 对领域内每一种可持久创建对象验证修改与删除；不可变历史必须明确标为例外，不能靠遗漏按钮形成事实上的只增不减。
13. 同步 README、架构、开发说明和模块文档；只有实现并验证的行为才能写成已交付能力。

上面的 13 步是实施清单；“标准生产流水线”是阶段门。前者用于安排开发任务，后者用于判断一个阶段是否真的完成，两者必须同时满足。

## 第二个特化模式前的注册表门槛

当前文献智库是第一套参考实现，部分入口、导航、项目筛选和领域分发仍显式使用 `literature-library`、`/literature/` 与 `literature` 工具画像。它可以支撑第一套模式，但不应通过继续复制 `if literature / if analysis / if planning` 来增加后续模式。

在开发第二个特化模式前，应先实现统一注册表。此项目前是架构门槛，不是已经交付的能力。

后端模式定义至少包含：

```text
project_type
supported_schema_versions
tool_profile
project detector / validator / initializer
tool schema factory / executor mapping
prompt context builder
projection router or service binding
```

前端模式定义至少包含：

```text
id
title / description / icon
project_type
entry_url
project initializer / launcher
navigation return target
required skin slots
```

功能大厅从前端注册表渲染入口；项目识别、工具白名单和 Prompt 分发从后端注册表取得定义。领域内部仍可以拥有专属 routes、service 和前端目录，但共享内核不再逐个追加模式分支。

注册表的目标是减少分散条件判断，不是把所有模式强迫成同一种业务 schema。项目结构、领域工具和投影 UI 仍然由各模式独立定义。

## 经验红线

- 不先做一个长期存在的独立聊天 Demo，再尝试接回 Workmode；静态布局样稿必须可随时丢弃。
- 不复制 session、JSONL、上下文、压缩、停止、设置、主题或认证。
- 不把 Prompt 当成路径、字段和固定目录的安全边界。
- 不把前端勾选状态、关键词确认或 AI 文字当成写入权限。
- 不把工具结果或前端 React state 当成第二份权威业务数据。
- 不用后端兜底偷偷执行模型没有调用的领域工具。
- 不在真实样例、失败路径和桌面安装包尚未验证时宣称“功能上线即可用”。
- 不把皮肤适配推迟到模式发布以后；新模式必须与功能大厅、工作台和既有皮肤体系共同验收。

## 领域写入原则

- 模型调用一个写工具就代表请求执行该领域操作，不额外增加 proposal/confirm/reject 或关键词审批状态机。
- “直接执行”不等于无边界：项目身份、相对路径、固定结构、字段 schema、引用关系、原子替换和冲突检测仍由后端负责。
- 用户当前选中的文件只用于提高相关性；权限来自“对象属于当前项目且操作满足领域策略”。
- 工具成功结果应列出 `operation`、目标 ID、`changed_files` 和 `updated_fields`；失败结果应有稳定 `error_code`、消息和空的 `changed_files`。
- AI 文字不是写入凭证。前端只根据正式工具事件和重新读取的权威文件显示成功状态。

## 发布门槛

一个特化模块在进入桌面导航或 Release 前至少需要：

- 固定结构初始化与版本识别测试；
- 工具画像中不存在越权通用工具；
- 写入、冲突、损坏文件和路径越界回归；
- session、工具卡、停止和压缩复用验证；
- 应用级入口、返回导航、流式正文/工具顺序与共享外观状态验证；
- 基础暗色、基础亮色、高对比、Neon 和至少一套官方签名皮肤的跨页面验证；
- 前端无静态假数据或隐形第二状态源；
- 一个真实项目的端到端验收记录；
- README 与相关 `docs/` 已同步。
