# 皮肤实验室迁移矩阵

本文记录 `design/skin-lab/` 静态样稿迁入 0.7.0 官方签名皮肤系统的状态。静态 HTML 只用于比较视觉，不是可导入代码；正式皮肤必须在真实 Workmode 组件上实现，并由官方审核、签名和回归。

## 新协议基线

Workmode 核心维护项目、文件、会话、消息、工具结果、上下文、输入区、查看器、设置和状态栏，并通过稳定的 `data-skin-slot` 暴露这些真实内容。每套正式皮肤包含：

- `layout.css`：决定槽位排列、尺寸、滚动边界和响应式降级；
- `visual.css`：决定配色、字体、材质、边框、图标、背景、装饰和动画；
- `manifest.json`：声明 ID、版本、基础明暗和本地资源；
- `signature.json`：覆盖上述文件及全部素材的官方 Ed25519 签名。

协议允许官方皮肤写完整 CSS，因此不再受旧积木枚举限制；代价是只接受官方签名包。皮肤不能携带 JavaScript、HTML、远程资源或权限，也不能创建虚构项目、工具结果和遥测。

## 问题—影响—修复成本

| 问题 | 影响 | 修复成本 |
| --- | --- | --- |
| 七套奖励皮肤均已完成独立 CSS，但缺少完整真实应用截图矩阵 | 自动契约能阻止结构退化，仍不能代替不同分辨率下的人工视觉验收 | 中：逐套补空会话、Markdown、工具状态和文件查看器截图 |
| CSS 可以遮挡设置或破坏小窗口布局 | 官方签名也不能自动保证视觉无 bug | 中：签名前执行设置可达、滚动、窗口缩放和紧急恢复基线 |
| 奖励签名包若混入公开发行产物 | 原始测试奖励失去边界，Release 体积和许可审计也变复杂 | 低：`packages/` 固定 Git 忽略，构建契约禁止读取皮肤库，CI 只上传应用产物 |
| 静态样稿缺少真实长文档、工具失败和设置页面 | 只看 demo 容易高估迁移完成度 | 中：每套至少保存四类真实工作区截图并人工检查 |
| 签名密钥丢失或泄露 | 无法延续官方身份，或攻击者可签恶意 CSS | 高：离线备份；泄露后发布新公钥/keyId 和新应用版本 |

## 迁移纪律

1. 先选一套样稿，明确它对真实项目、文件、会话、消息、工具、设置和查看器的布局，而不是只换颜色。
2. 在 `skin-library/sources/<skin-id>/` 编写独立 `layout.css` 和 `visual.css`；所有素材先确认来源与许可证。
3. 在真实 Workmode 上检查窄窗口、长 Markdown/PDF、消息流、工具运行/失败/取消、设置页和输入框。
4. 运行自动测试后用 `scripts/build-skin-library.ps1` 净化并签包；不得手工修改 `skin-library/packages/` 中已经签名的 ZIP。
5. 全部签名包只保留在被 Git 忽略的 `skin-library/packages/` 并由维护者手动发放；GitHub Release 只发布应用本体。
